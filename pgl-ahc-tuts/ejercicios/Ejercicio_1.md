Ejercicio 1

Implementé una pantalla de register, con tres inputs y un botón que cuando es pulsado el botón activa un evento que lanza una función asíncrona que devuelve una promesa, esa función recoge lo que está en los inputs, "Nombre", "correo" y una "contraseña", más tarde llama a otra clase que valida que sea fuerte, es decir, que su longitud sea mayor a 8 caracteres, una mayúscula y un número, en caso de que pase las validaciones, se llama a una función asíncrona que guarda en un objeto, a esa función le pasan los parámetros del nombre, el correo y la contraseña, esa función viene de la clase "auth.service", esta clase se encarga de preparar un cuerpo con los parámetros que enviamos desde la página register, "nombre", "correo" y "contraseña", los mete dentro de un objeto dentro de una función "authService" y lo envía junto al método http que va a utilizar para ser fetch a la api, en este caso un post porque vamos a enviarle un header y el endpoint, el resultado de la función al hacer la consulta a la api, devolverá un tipo ApiResponse con un mensaje de la api si lo tiene, el código de solicitud para decirte si fue realizado correctamente o si te devuelve un objeto o nulo.

La función "authService" que está en la clase "auth.Service" llama a la función asíncrona "httpRequest" dentro de la clase http.service, aquí se encuentra la parte importante de comunicación con la api. Realicé esta clase con la función universal que encapsula el fetch a la api porque es una manera más limpia y coherente a la hora de realizar peticiones a la api, no tenemos que crear fetchs en toda la app y repetir código de manera innecesaria cada vez que debamos mandar un header o realizar una consulta, a parte, es un requirimiento del propio ejercicio, aunque sea algo natural de hacerse de manera no intencional porque el programa por las complegidades que conlleva es más fácil y práctico hacerlo así, primero se importa la ip del equipo donde está escuchando la api y se exporta los verbos para hacer peticiones a la api, en este caso solo "POST" y "GET", porque solo buscamos enviar el cuerpo de el registro de un usuario con nombre, email y contraseña. Más adelante se encuentra la función "httpReques", esta función al igual que el "authService" son asíncronas porque las peticiones no se realizan de manera inmediata, entonces se pausa esa ejecución de la función hasta que no se complete. La función recibe el endpoint por "path", el verbo en este caso "POST", un cuerpo que buscamos que pueda ser opcional ya que podemos hacer peticiones de "POST" para mandar un cuerpo de una solicitud o no si buscamos hacer un "GET" a la pantalla de "welcome", por último un header opcional que servirá para el segundo ejercicio cuando toque usarse tokens, vendrá con dos string, la parte inicial con "Bearer" y la llave cifrada y hasheada del token, pero este primer ejercicio no lo requiere porque no usamos tokens. La función es introducida dentro de un objeto y hacemos fetch a la api con la ip, el endpoint y el verbo, hacemos una condicional que si hay el header extra se manda también con el token o si no hay no se manda, por último si el cuerpo es distinto de nulo, lo convertimos a JSON para enviarlo a la api, si es nulo es indefinido y no se manda. Una vez recibida la respuesta de la api se guarda en objeto como resultado de ejecución del fetch, por último ese objeto comprobamos si tiene contenido con ("content-type) o si es nula para evitar errores y guardamos el resultado en una variable, luego utilizaremos esa variable con el resultado obtenido para saber si ese resultado que envió la api que puede ser json o venir con otra cosa. Por último se decide si la respuesta es JSON se transforma a un objeto tipo JSON o si es texto se transforma en string porque la api puede devolver un mensaje de texto también porque el método debe saber gestionar la respuesta de la api y convertirla. Por último de esa respuesta el programa comprueba si hubo una respuesta correcta, si hay respuesta correcta o no es correcta, crea el cuerpo de la respuesta de la api, si es un mensaje con código de error lo muestra o si es un array de mensajes de error los une, si no hay mensaje lanza el código de error http o directamente lanza un error por mensaje completo, en caso de que no haya habido errores retorna el objeto donde guarda la respuesta de la api como el objeto que retorna la promesa, que a su vez guardará "autService" y que usará la pantalla register para saber si recibe un mensaje correcto de la api, remplaza la ruta por el Login y te redirigirá a ella o si en cambio recibe una excepción, lanza un mensaje con el error equivalente.
